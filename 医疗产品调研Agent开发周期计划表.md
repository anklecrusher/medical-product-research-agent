# 医疗产品调研 Agent 开发周期计划表

## 目标说明

本计划不按真实日期排期，而是按项目从 0 到 1 的技术流程拆解：哪些模块必须先做，哪些可以并行做，哪些必须等待前置输入完成后才能继续。

项目目标是构建一个面向医疗产品经理的调研 Agent / Workflow：输入一句调研需求，系统自动检索论文、厂商资料、监管资料和本地上传资料，抽取证据，生成结构化 Markdown 报告并导出 PDF。

## 总体技术流程

```text
需求输入
  ↓
任务解析与调研规划
  ↓
多源检索
  ├─ 论文检索
  ├─ 厂商/市场资料检索
  ├─ 监管资料检索
  └─ 本地上传资料检索
  ↓
资料抓取与解析
  ↓
证据抽取与结构化
  ↓
证据去重、冲突检查、引用管理
  ↓
报告大纲生成
  ↓
章节写作
  ↓
事实核查与引用核查
  ↓
Markdown 报告
  ↓
PDF 渲染
  ↓
Web 工作台与任务管理
```

## 必须单独优先完成的模块

这些模块是项目地基，必须先独立完成，后续模块都依赖它们。

| 模块 | 为什么必须先做 |
|---|---|
| 项目基础骨架 | 决定目录结构、配置、日志、输出文件位置，后续所有模块都会接入 |
| 数据结构定义 | 决定任务、来源、证据、参数和报告之间如何传递，避免后续 LLM 输出失控 |
| LangGraph 工作流骨架 | 决定整体流程如何编排、失败如何重试、中间状态如何保存 |
| LLM 调用封装 | 所有 agent 都会调用 LLM，必须统一模型、重试、结构化输出和日志 |
| 报告模板定义 | 决定最终交付长什么样，影响证据抽取和报告写作字段设计 |

## 可以并行开发的模块

这些模块之间耦合较低，只要统一 `SourceRecord` 和 `EvidenceItem` 等数据结构，就可以并行推进。

| 并行组 | 可并行模块 | 前置条件 |
|---|---|---|
| 检索 Connector 组 | 论文检索、厂商资料检索、监管资料检索 | 数据结构定义完成 |
| 解析组 | 网页解析、PDF 解析、元数据解析、来源快照保存 | 项目骨架和来源结构完成 |
| 抽取组 | 文献证据抽取、产品参数抽取、监管信息抽取 | LLM 封装和解析结果格式完成 |
| 存储组 | SQLite 任务表、来源表、证据表、报告表 | 数据结构定义完成 |
| 交付组 | Markdown 模板、HTML 样式、PDF 渲染样式 | 报告模板方向确定 |
| Web 组 | FastAPI 接口、简单页面、文件上传接口 | CLI 核心流程初步可用 |

## 必须等待前置输入的模块

这些模块不能太早做，否则容易返工。

| 模块 | 必须等待什么 | 原因 |
|---|---|---|
| 证据抽取 Agent | 等资料解析输出稳定 | 如果解析字段变动，抽取 prompt 和 schema 会反复改 |
| 报告大纲生成 | 等证据类型和报告模板稳定 | 大纲要根据实际证据密度和报告样式决定 |
| 章节写作 Agent | 等证据清洗和冲突检查完成 | 否则容易把未核查信息写成确定结论 |
| 引用核查 | 等报告初稿完成 | 核查对象是报告里的 claim 和参数表 |
| PDF 渲染 | 等 Markdown 结构稳定 | 表格和引用格式未定时做 PDF 样式会返工 |
| React 工作台 | 等 API 和 workflow 稳定 | 前端依赖任务状态、报告结构和人工确认逻辑 |
| 权限系统细化 | 等上传资料和 LLM 调用链路明确 | 需要知道哪些内容可能出本地、在哪一步出本地 |

## 推荐开发顺序

### 第一轮：跑通最小闭环

目标是先不要做完整前端，只验证核心能力。

```text
项目骨架
  ↓
数据结构
  ↓
LLM 封装
  ↓
LangGraph 骨架
  ↓
论文/网页检索最小版本
  ↓
网页/PDF 解析最小版本
  ↓
证据抽取最小版本
  ↓
报告 Markdown 生成
  ↓
PDF 导出
  ↓
CLI 一句话运行
```

第一轮验收输出：

```text
outputs/{task_id}/sources.json
outputs/{task_id}/evidence.json
outputs/{task_id}/report.md
outputs/{task_id}/report.pdf
```

### 第二轮：增强调研质量

目标是让报告更接近真实产品经理可用的调研草稿。

```text
增加 PubMed / Semantic Scholar / Crossref
  ↓
增加 FDA / NICE / 厂商 PDF
  ↓
增加产品参数抽取
  ↓
增加证据去重
  ↓
增加参数冲突检查
  ↓
增加引用核查
  ↓
用 DBS/SCS 主题做回归
```

第二轮验收重点：

- 参数表是否完整。
- 每个关键参数是否有来源。
- 不同来源冲突是否被标记。
- 报告风格是否接近现有 DBS/SCS 草稿。

### 第三轮：做内网可用 MVP

目标是让非技术用户可以使用。

```text
FastAPI 后端
  ↓
任务创建 API
  ↓
任务状态 API
  ↓
文件上传 API
  ↓
报告下载 API
  ↓
简单 HTML 页面
```

第三轮验收重点：

- 用户能在页面输入需求。
- 用户能上传参考资料。
- 用户能看到任务状态。
- 用户能下载 Markdown/PDF。

### 第四轮：产品化工作台

目标是提升团队使用体验。

```text
React/Vite 前端
  ↓
任务列表
  ↓
实时日志
  ↓
报告预览
  ↓
证据表格
  ↓
人工确认/驳回来源
  ↓
报告版本管理
```

第四轮验收重点：

- 产品经理可以检查证据。
- 可以发现未核查结论。
- 可以管理历史报告。
- 可以编辑或重生成某个章节。

## 主要技术模块与建议

| 技术模块 | 推荐技术 | 作用 |
|---|---|---|
| 编排 | LangGraph | 控制 workflow 节点、状态、重试、人工介入 |
| 后端 | FastAPI | 提供任务、上传、状态、下载接口 |
| LLM | OpenAI SDK 或兼容 API | 需求理解、证据抽取、报告写作、核查 |
| 检索 | httpx + 各数据源 API | 获取论文、厂商、监管资料 |
| 网页解析 | trafilatura / BeautifulSoup | 抽取网页正文和元数据 |
| PDF 解析 | PyMuPDF / pypdf | 提取 PDF 文本和页级内容 |
| Schema | Pydantic | 约束 agent 输出，降低幻觉 |
| 存储 | SQLite 起步，后续 PostgreSQL | 保存任务、证据、来源、报告 |
| 本地资料索引 | LlamaIndex / Chroma 可选 | 检索用户上传资料 |
| 报告模板 | Markdown + Jinja2 | 稳定生成长报告 |
| PDF | Playwright Chromium | 将 HTML/Markdown 渲染为 PDF |
| 前端 | 先简单 HTML，后 React/Vite | 从 MVP 页面升级到产品化工作台 |

## 多线程协作与参考项目管理

如果后续使用多个线程并行开发，仓库必须保留跨线程上下文，避免每个线程重复理解项目目标或遗忘关键参考项目。

建议新增并长期维护两个文件：

| 文件 | 作用 | 更新时机 |
|---|---|---|
| `AGENTS.md` | 给所有新线程看的项目说明、架构方向、参考项目边界、隐私规则和开发规范 | 架构、流程、技术选型或协作规则变化时更新 |
| `LOG.md` | 记录各线程做过什么、改了哪些文件、做了哪些决定、还有哪些后续事项 | 每个线程完成一次有意义变更后追加 |

三个 GitHub 参考项目应写入 `AGENTS.md`，并按模块触发参考：

| 参考项目 | 什么时候参考 | 参考重点 |
|---|---|---|
| `Future-House/paper-qa` | 做论文问答、文献证据、引用溯源、文献 chunk 和 citation 逻辑时 | 科学文献如何检索、回答和带引用输出 |
| `assafelovic/gpt-researcher` | 做一句需求拆解、多源搜索、深度调研流程、资料汇总时 | 如何规划搜索任务、抓取多源资料并形成调研草稿 |
| `stanford-oval/storm` | 做长报告大纲、章节规划、多段落研究报告组织时 | 如何先规划报告结构，再分章节生成长文 |

协作规则：

- 新线程开始前先读 `AGENTS.md`、本计划表和 `LOG.md` 最新记录。
- 每个线程只改自己负责的模块，避免顺手重构无关部分。
- 如果改动影响 schema、workflow、存储、报告格式或隐私策略，必须同步更新文档。
- 每次完成后在 `LOG.md` 追加变更摘要、关键决定和后续事项。

## 风险与控制点

| 风险 | 控制方式 |
|---|---|
| LLM 编造参数 | 参数必须来自结构化证据，缺来源标记 `needs_review` |
| 来源质量不稳定 | 区分论文、厂商、监管、内部资料来源类型 |
| PDF 解析失败 | 保存原文链接和解析日志，允许人工上传或补充 |
| 报告太泛 | 固定医疗产品调研模板，强制输出参数表和证据表 |
| 内部资料泄露 | 内部来源默认本地处理，外部 LLM 调用前做权限检查 |
| 前端过早复杂化 | 先 CLI 和 FastAPI 简单页面，等流程稳定后再做 React |

## 第一阶段建议完成标准

第一阶段完成时，项目应至少支持：

- 输入一句调研需求。
- 自动生成检索计划。
- 自动检索至少 2 类公开来源。
- 自动抽取结构化证据。
- 自动生成 Markdown 报告。
- 自动导出 PDF。
- 保存来源列表和证据 JSON。
- 能用 DBS/SCS 类主题跑通一次。

第一阶段暂不强制支持：

- 完整 React 前端。
- 多用户权限。
- 团队协作。
- 完整本地知识库。
- 所有医学数据库。
- 复杂图表自动生成。
